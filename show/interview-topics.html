一、IOC & DI 容器（核心考点）

📌 高频面试题 1：什么是 IOC、DI？你在 LightSSM 中是如何实现的？

💡 满分回答模板（两段式）：
① 原生 Spring 原理：
IOC（Inversion of Control）控制反转，将对象的创建和依赖关系管理交给 Spring 容器，而不是代码中 new 对象。DI（Dependency Injection）依赖注入，是 IOC 的具体实现方式，通过反射将依赖的对象注入到目标对象中。

② 我的 LightSSM 实现：
在我的 LightSSM 中，我实现了一个精简版的 IOC 容器：
// LightSSM 简易 IOC 容器 public class LightIOC { // 单例 Bean 容器 private Map<String, Object> beanMap = new ConcurrentHashMap<>(); // 1. 包扫描，注册 Bean public void scanPackage(String packagePath) { Set<Class?> classes = getClasses(packagePath); for (Class? clazz : classes) { if (clazz.isAnnotationPresent(MyComponent.class)) { String beanName = getBeanName(clazz); beanMap.put(beanName, clazz); } } } // 2. 获取 Bean（懒加载） public Object getBean(String beanName) throws Exception { Object instance = beanMap.get(beanName); if (instance == null) { Class? clazz = (Class?) beanMap.get(beanName); instance = clazz.getDeclaredConstructor().newInstance(); beanMap.put(beanName, instance); // 3. 依赖注入 injectDependencies(instance); } return instance; } // 4. 依赖注入（字段注入） private void injectDependencies(Object obj) { Field[] fields = obj.getClass().getDeclaredFields(); for (Field field : fields) { if (field.isAnnotationPresent(MyAutowired.class)) { field.setAccessible(true); Object bean = getBean(getBeanName(field.getType())); field.set(obj, bean); } } } }
核心差异：原生 Spring 有完整的 BeanDefinition、三级缓存、BeanPostProcessor 等扩展机制，我的 LightSSM 只保留了最核心的 IOC+DI 骨架，用于理解 Spring 设计思想。

📌 高频面试题 2：Spring 三级缓存、循环依赖，你 LightSSM 为什么没有实现？

💡 满分回答模板：
① 原生 Spring 三级缓存机制：
Spring 通过三级缓存解决单例 Bean 的 setter 注入循环依赖：
flowchart LR A["getBean(A)"] --> B["一级缓存 singletonObjects"] B --> C{"命中？"} C -->|否 | D["实例化 A"] D --> E["提前暴露到三级缓存"] E --> F["属性注入 B"] F --> G["getBean(B)"] G --> H["实例化 B"] H --> I["属性注入 A"] I --> J["从三级缓存拿 A 的早期引用"] J --> K["B 完成创建"] K --> L["A 完成创建"] L --> M["放入一级缓存"]
// Spring 三级缓存结构 singletonObjects // 一级：成品 Bean earlySingletonObjects // 二级：半成品早期引用 singletonFactories // 三级：对象工厂（可延迟 AOP 代理）
② 我的 LightSSM 为什么没实现：

🎯 设计取舍（面试加分点）：

  • 定位差异：LightSSM 是学习型框架，核心目的是理解 IOC、DI、MVC 骨架，循环依赖属于高级复杂特性
  • 实现成本：三级缓存 + 早期引用暴露 + AOP 代理延迟创建，实现成本高，代码量会增加 2-3 倍
  • 业务场景：日常 90% 的业务场景不会出现循环依赖，良好的设计应该避免循环依赖
  • 学习价值：理解 Spring 为什么要用三级缓存（支持 AOP 代理），比实现本身更重要
话术总结："我的 LightSSM 虽然没有实现三级缓存，但我通过研读 Spring 源码，理解了它的设计思想：通过提前暴露早期引用来打破循环，通过工厂模式延迟 AOP 代理的创建。这让我对 Spring 的架构设计有了更深的理解。"

二、AOP & 声明式事务（理解差异）

📌 高频面试题 1：Spring AOP 底层原理？你 LightSSM 为什么没有 AOP？

💡 满分回答模板：
① 原生 Spring AOP 原理：
Spring AOP 基于动态代理实现，有两种方式：
  • JDK 动态代理：基于接口实现，使用 InvocationHandler，要求目标类实现接口
  • CGLIB 代理：基于子类实现，使用 MethodInterceptor，可以代理没有接口的类
  • 选择策略：有接口优先用 JDK，没有接口用 CGLIB（SpringBoot 2.x 后默认 CGLIB）
② 我的 LightSSM 为什么没有 AOP：

🎯 设计考量：

  • 复杂度：AOP 是 Spring 第二大核心，涉及动态代理、拦截器链、切面织入等，实现复杂度远高于 IOC
  • 定位：LightSSM 的核心目标是吃透 IOC+MVC 骨架，AOP 可以单独作为扩展模块学习
  • 事务支撑：声明式事务基于 AOP，没有 AOP 的情况下，我使用 MyBatis 原生事务管理
// 我的 LightSSM 手动事务管理 try (SqlSession session = sqlSessionFactory.openSession(false)) { // 1. 获取 Mapper UserMapper mapper = session.getMapper(UserMapper.class); // 2. 执行业务 mapper.insert(user); mapper.insert(role); // 3. 手动提交事务 session.commit(); } catch (Exception e) { // 4. 回滚事务 session.rollback(); }

三、SpringMVC（你的核心复刻模块，重中之重）

📌 高频面试题 1：原生 SpringMVC 完整流程 vs 你的 LightSSM 实现

📘 原生 SpringMVC（完整版） 🔶 我的 LightSSM（精简版）
五大核心组件：
1. DispatcherServlet（前端控制器）
2. HandlerMapping（处理器映射）
3. HandlerAdapter（处理器适配器）
4. ViewResolver（视图解析器）
5. HandlerInterceptor（拦截器）

完整流程：
请求 → DispatcherServlet → HandlerMapping → HandlerAdapter → Controller → ModelAndView → ViewResolver → 视图渲染 → 响应 		精简组件：
1. 自定义 DispatcherServlet
2. 简易 HandlerMapping（URL 映射表）
3. 直接反射执行

简化流程：
请求 → DispatcherServlet → 匹配 URL → 反射执行 Controller 方法 → 响应

📌 高频面试题 2：你手写 DispatcherServlet 的实现逻辑

💡 满分回答 + 源码：
我的 LightSSM 实现：
// LightSSM 自定义前端控制器 @WebServlet("/") public class LightDispatcherServlet extends HttpServlet { // URL 到方法的映射表 private Map<String, Handler> handlerMap = new ConcurrentHashMap<>(); // 1. 初始化：扫描@Controller，构建映射表 public void init() { Map<String, Object> controllers = getBeansWithAnnotation(MyController.class); for (Entry<String, Object> entry : controllers.entrySet()) { Object controller = entry.getValue(); // 扫描@RequestMapping 注解 Method[] methods = controller.getClass().getMethods(); for (Method method : methods) { if (method.isAnnotationPresent(MyRequestMapping.class)) { MyRequestMapping mapping = method.getAnnotation(MyRequestMapping.class); String url = mapping.value(); handlerMap.put(url, new Handler(controller, method)); } } } } // 2. 处理请求 protected void service(HttpServletRequest req, HttpServletResponse resp) { String url = req.getRequestURI(); Handler handler = handlerMap.get(url); if (handler == null) { resp.setStatus(404); return; } // 3. 反射执行 Controller 方法 try { handler.method.invoke(handler.controller, req, resp); } catch (Exception e) { e.printStackTrace(); } } // 内部类：处理器 class Handler { Object controller; Method method; Handler(Object c, Method m) { controller = c; method = m; } } }
核心差异：原生 SpringMVC 有 HandlerAdapter 适配器层（支持多种 Handler 类型）、视图解析器（支持 JSP、Thymeleaf 等）、拦截器链。我的 LightSSM 只保留了最核心的"URL 映射 + 反射执行"，用于理解 SpringMVC 的本质。

五、框架设计深挖题（面试官专属追问）

📌 高频面试题：你做 LightSSM 的初衷是什么？简化了哪些冗余设计？

💡 满分回答模板：
① 初衷：
学习 Spring 源码时，发现代码量太大（几十万行），直接阅读源码很难抓住核心。所以我决定自己手写一个精简版，只保留最核心的 IOC+DI+MVC 骨架，通过实践来理解 Spring 的设计思想。

② 简化的设计：
  • BeanDefinition 层：原生 Spring 用 BeanDefinition 封装 Bean 元信息，我直接用 Class 对象存储
  • 三级缓存：原生 Spring 用三级缓存解决循环依赖，我直接舍弃（理由：学习重点不在此）
  • BeanPostProcessor：原生 Spring 有大量扩展点，我全部舍弃
  • AOP 模块：原生 Spring 基于动态代理实现 AOP，我完全没实现（复杂度太高）
  • 视图解析器：原生 SpringMVC 支持多种视图技术，我只做 RESTful API，不需要视图解析
  • 拦截器链：原生 SpringMVC 有拦截器链，我没实现（核心是理解请求分发）
③ 收获：
通过手写 LightSSM，我理解了 Spring 最本质的设计思想：IOC 容器负责托管对象，DI 自动注入依赖，MVC 统一分发请求。虽然我的框架功能简单，但让我看 Spring 源码时不再迷茫，知道哪些是核心、哪些是扩展。
