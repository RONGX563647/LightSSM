核心概念解析

1. 循环依赖问题

问题场景

当 BeanA 依赖 BeanB，BeanB 又依赖 BeanA 时，如何避免无限递归创建？三级缓存机制是经典解决方案。

2. 三级缓存结构

  • 一级缓存：singletonObjects - 存储完整的 Bean 实例
  • 二级缓存：earlySingletonObjects - 存储早期 Bean 引用
  • 三级缓存：singletonFactories - 存储 Bean 工厂

3. AOP 代理机制

两种代理方式

  • JDK 动态代理：基于接口实现
  • CGLIB 代理：基于子类实现

选择策略：优先使用 JDK 代理，否则使用 CGLIB。

4. 动态 SQL 解析

SqlNode 树结构

  • MixedSqlNode：混合节点
  • IfSqlNode：条件节点
  • TextSqlNode：文本节点

1. SqlNode 接口设计

// SqlNode 接口 interface SqlNode { apply(context) } // MixedSqlNode - 混合节点 class MixedSqlNode implements SqlNode { contents = List<SqlNode> apply(context) { for node in contents: node.apply(context) } } // IfSqlNode - 条件节点 class IfSqlNode implements SqlNode { test = String contents = SqlNode apply(context) { if evaluator.evaluateBoolean(test, context.getBindings()): contents.apply(context) } } // TextSqlNode - 文本节点 class TextSqlNode implements SqlNode { text = String apply(context) { parser = new GenericTokenParser("#{", "}", token -> context.getBindings().get(token)) context.appendSql(parser.parse(text)) } }

2. 插件机制

// 性能监控插件 class PerformanceInterceptor implements Interceptor { intercept(invocation) { start = System.currentTimeMillis() try { return invocation.proceed() } finally { elapsed = System.currentTimeMillis() - start logger.info("SQL 执行耗时：{}ms", elapsed) } } plugin(target) { return Plugin.wrap(target, this) } } // 插件包装逻辑 class Plugin { static wrap(target, interceptor) { signatureMap = getSignatureMap(interceptor) if shouldWrap(target.getClass(), signatureMap): return Proxy.newProxyInstance( target.getClass().getInterfaces(), new PluginInvocationHandler(target, interceptor) ) return target } }

3. 源码对比：JDK 动态代理

原生 Spring - JdkDynamicAopProxy
final class JdkDynamicAopProxy implements AopProxy, InvocationHandler { private final AdvisedSupport advised; public Object invoke( Object proxy, Method method, Object[] args) { // 1. 获取拦截器链 List<MethodInterceptor> interceptors = advised .getInterceptorsAndDynamicMethodMatchers( method, targetClass); // 2. 创建调用链 MethodInvocation invocation = new ReflectiveMethodInvocation( proxy, target, method, args, targetClass, interceptors); // 3. 执行调用链 return invocation.proceed(); } }
LightSSM - JdkDynamicAopProxy
public class JdkDynamicAopProxy implements InvocationHandler { private Object target; private List<Advisor> advisors; public Object invoke( Object proxy, Method method, Object[] args) { // 1. 查找匹配的 Advisor List<Advisor> matched = findMatchedAdvisors(method); // 2. 执行 Before 通知 for (Advisor advisor : matched) { advisor.getBeforeAdvice() .before(method, args); } // 3. 执行目标方法 Object result = method .invoke(target, args); // 4. 执行 AfterReturning 通知 for (Advisor advisor : matched) { advisor.getAfterAdvice() .afterReturning(result); } return result; } }

💡 AOP 本质洞察

AOP 的核心就三件事：1. 在哪里拦截（Pointcut + AspectJ 表达式），2. 用什么代理（JDK 接口代理 vs CGLIB 子类代理），3. 执行什么逻辑（Before/After/Around 通知）。原生 Spring 的复杂性在于支持多种通知类型、Advisor 链管理、暴露连接点信息等高级特性。LightSSM 剥离了这些复杂性，让你看到 AOP 的骨架：代理 + 拦截 = AOP。

🎯 面试核心：AOP 的本质

  • AOP 的本质：面向切面编程，将横切关注点（日志、事务、权限）与业务逻辑分离
  • 动态代理的本质：在运行时创建目标对象的代理，在方法调用前后织入通知
  • JDK 代理 vs CGLIB：JDK 基于接口（InvocationHandler），CGLIB 基于继承（MethodInterceptor）
  • Advisor 链的执行顺序：按照 order 排序，Before 正序执行，After 逆序执行
  • Around 通知的威力：可以控制是否执行目标方法，可以修改返回值
