ORM 组件职责

SqlSession

职责：数据库会话门面，提供统一的数据库操作接口。

类比：类似于银行的柜台窗口，客户通过窗口办理各种业务。

Executor

职责：SQL 执行器，负责 SQL 的实际执行，包括查询、更新、事务管理等。

实现：BaseExecutor 使用模板方法模式定义执行流程。

StatementHandler

职责：SQL 语句处理器，负责创建 PreparedStatement、设置参数、执行查询。

1. Executor 模板方法

// 模板方法：定义查询流程 class BaseExecutor { query(ms, parameter, rowBounds) { // 1. 获取 BoundSql boundSql = ms.getBoundSql(parameter) // 2. 创建 CacheKey（一级缓存） key = createCacheKey(ms, parameter, rowBounds, boundSql) // 3. 查询本地缓存 cachedResult = localCache.get(key) if cachedResult != null: return cachedResult // 4. 执行数据库查询（子类实现） result = doQuery(ms, parameter, rowBounds, boundSql) // 5. 缓存结果 localCache.put(key, result) return result } abstract doQuery(ms, parameter, rowBounds, boundSql) } // 简单执行器实现 class SimpleExecutor extends BaseExecutor { doQuery(ms, parameter, rowBounds, boundSql) { result = new List() conn = transaction.getConnection() sh = newStatementHandler(ms, boundSql) stmt = sh.prepare(conn) sh.parameterize(stmt) result = sh.query(stmt) return result } }

2. Mapper 代理机制

// Mapper 代理：动态代理实现 class MapperProxy implements InvocationHandler { sqlSession = SqlSession mapperInterface = Class methodCache = Map<Method, MapperMethod> invoke(proxy, method, args) { if method.declaringClass == Object.class: return method.invoke(this, args) mapperMethod = methodCache.computeIfAbsent(method) return mapperMethod.execute(sqlSession, args) } } class MapperMethod { execute(sqlSession, args) { switch (command.type) { case SELECT: if method.returnsList(): return sqlSession.selectList(command.name, args) else: return sqlSession.selectOne(command.name, args) case INSERT: return sqlSession.insert(command.name, args) case UPDATE: return sqlSession.update(command.name, args) case DELETE: return sqlSession.delete(command.name, args) } } }

3. 结果集映射

// 结果集处理器：ResultSet -> Java 对象 class DefaultResultSetHandler { handleResultSets(stmt) { result = new List() rs = stmt.getResultSet() while rs.next(): resultObject = createResultObject(rs) applyAutomaticMappings(resultObject, rs) applyPropertyMappings(resultObject, rs) result.add(resultObject) return result } applyAutomaticMappings(obj, rs) { meta = rs.getMetaData() for i in 1..meta.getColumnCount(): columnName = meta.getColumnName(i) propertyName = camelCase(columnName) th = typeHandlerRegistry.getTypeHandler(rs.getObject(columnName).class) value = th.getResult(rs, columnName) setProperty(obj, propertyName, value) } }

3. 源码对比：Executor 执行器

原生 MyBatis - BaseExecutor
public abstract class BaseExecutor implements Executor { // 一级缓存 protected PermittedCache cache; // 事务管理 protected Transaction transaction; public List<E> query( MappedStatement ms, Object parameter) { // 1. 创建 CacheKey CacheKey key = createCacheKey(...); // 2. 从一级缓存查找 List<E> list = (List<E>) cache .getObject(key); if (list == null) { // 3. 缓存未命中，查询数据库 list = queryFromDatabase( ms, parameter, key); } return list; } protected abstract List<E> queryFromDatabase(...); }
LightSSM - SimpleExecutor
public class SimpleExecutor implements Executor { private Configuration configuration; private Connection connection; public List<E> query( MappedStatement ms, Object parameter) { // 1. 获取 SQL String sql = ms.getSql(); // 2. 创建 PreparedStatement PreparedStatement stmt = connection .prepareStatement(sql); // 3. 设置参数 setParameters(stmt, parameter); // 4. 执行查询 ResultSet rs = stmt .executeQuery(); // 5. 处理结果集 return handleResultSet(rs); } }

💡 架构设计洞察

原生 MyBatis 的强大在于缓存机制和插件系统：一级缓存、二级缓存、插件拦截器等。而 LightSSM 的精髓在于理解本质：ORM 的核心就是"SQL 执行 + 结果集映射"。通过手写简化版，你能清晰看到 MyBatis 是如何将 JDBC 封装成 ORM 框架的。

4. MyBatis 整合到 Spring 的关键

📝 整合核心步骤

  1. 扫描包路径：找到所有 Mapper 接口（@Mapper 扫描）
  2. 创建 FactoryBean：为每个 Mapper 创建 MapperFactoryBean
  3. 创建代理对象：FactoryBean.getObject() 返回 JdkDynamicAopProxy 代理
  4. 注入 SqlSession：代理对象内部持有 SqlSession 引用
  5. 方法调用：调用 Mapper 方法时，InvocationHandler.invoke() 执行 SqlSession 方法

🎯 面试核心：MyBatis 的本质

  • ORM 的本质：将 SQL 执行和结果集映射封装成对象操作
  • Mapper 接口为什么不需要实现类？：使用 JDK 动态代理创建代理对象
  • 一级缓存的作用：同一个 SqlSession 中，相同 SQL 不重复查询
  • 二级缓存的作用：多个 SqlSession 共享缓存（需要手动开启）
  • 插件原理：基于责任链模式，拦截 Executor、StatementHandler 等方法
