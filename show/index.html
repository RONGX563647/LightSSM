所有面试考点锚定你的项目，不再空洞，面试官深挖你完全接得住。

功能特性

IOC/DI 容器

支持 @Component、@Autowired 注解扫描与依赖注入，基于三级缓存解决循环依赖问题，支持 @Scope 配置单例/原型模式

SpringMVC 核心

实现 DispatcherServlet、HandlerMapping、HandlerAdapter，支持 @RequestMapping、@ResponseBody、@RequestParam、@PathVariable

ORM 模块

支持 XML SQL 映射配置、动态 SQL 拼接（OGNL 表达式）、参数自动绑定与 ResultSet 映射、插件拦截器机制

AOP 切面

基于 JDK/CGLIB 双代理实现，支持 @Before、@After、@Around 通知织入，支持 AspectJ 表达式切入点

核心代码

public class DefaultListableBeanFactory {
  // 一级缓存：完整 Bean 实例
  private Map<String, Object> singletonObjects = new ConcurrentHashMap<>();

  // 二级缓存：早期 Bean 引用
  private Map<String, Object> earlySingletonObjects = new ConcurrentHashMap<>();

  // 三级缓存：Bean 工厂
  private Map<String, ObjectFactory<?>> singletonFactories = new ConcurrentHashMap<>();

  // 三级缓存查找逻辑
  protected Object getSingleton(String beanName) {
    Object singletonObject = this.singletonObjects.get(beanName);
    if (singletonObject == null && isSingletonCurrentlyInCreation(beanName)) {
      singletonObject = this.earlySingletonObjects.get(beanName);
      if (singletonObject == null) {
        ObjectFactory<?> factory = this.singletonFactories.get(beanName);
        if (factory != null) {
          singletonObject = factory.getObject();
          this.earlySingletonObjects.put(beanName, singletonObject);
          this.singletonFactories.remove(beanName);
        }
      }
    }
    return singletonObject;
  }
}

💡 核心洞察

LightSSM 的三级缓存实现与原生 Spring 几乎一致！这证明了 Spring 的设计精髓： 通过三级缓存打破循环依赖的关键在于提前暴露早期引用。 原生 Spring 的复杂性在于支持 AOP 代理的提前织入（通过 BeanPostProcessor）， 而 LightSSM 去除了这个复杂机制，保留了最核心的循环依赖解决方案。

🎯 面试加分点：三级缓存的本质

  • 一级缓存是目标：存储完整可用的 Bean
  • 二级缓存是过渡：存储早期引用，解决"我依赖你，你依赖我"的死锁
  • 三级缓存是延迟：通过工厂延迟创建早期引用，给 AOP 留出代理机会
  • 核心思想：提前暴露、延迟创建、缓存复用

💡 架构设计洞察

原生 Spring 的强大在于扩展性：通过 HandlerMapping、HandlerAdapter 的接口抽象， 支持多种请求映射方式和处理器类型。而 LightSSM 的精髓在于理解本质： MVC 的核心就是"URL → Controller 方法 → 响应"的映射过程。 通过手写简化版，你能清晰看到每个环节的作用，这是背十遍源码都达不到的理解深度。

🎯 面试核心：MyBatis 整合到 Spring 的关键

  • 问题：Mapper 接口没有实现类，如何被 Spring 管理？
  • 解决方案：使用 FactoryBean 创建 Mapper 代理对象
  • 核心流程：
    1. 扫描包路径，找到所有 Mapper 接口
    2. 为每个 Mapper 创建 MapperFactoryBean
    3. FactoryBean.getObject() 返回 JdkDynamicAopProxy 代理
    4. 代理对象注入 SqlSession，执行 SQL 时调用 SqlSession 方法

💡 AOP 本质洞察

AOP 的核心就三件事： 1. 在哪里拦截（Pointcut + AspectJ 表达式） 2. 用什么代理（JDK 接口代理 vs CGLIB 子类代理） 3. 执行什么逻辑（Before/After/Around 通知）

原生 Spring 的复杂性在于支持多种通知类型、Advisor 链管理、暴露连接点信息等高级特性。 LightSSM 剥离了这些复杂性，让你看到 AOP 的骨架：代理 + 拦截 = AOP。
